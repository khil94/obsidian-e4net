
## 협업 분야 리스트

- 일감 관리
- 일정 관리
- 문서 관리
- 의사소통


## 일감 관리

도입할 이유를 먼저 명확히 하자.

- 서로 간 업무의 진행도 파악
- 서로의 일감 상태, 양, 세부 사항 파악 용이
- 전체 일감의 상태 파악
- 일감의 실질 working day 파악

도입하지 않을 이유

- 팀 규모가 소규모
- 서로 공유하는 작업이 많지 않다.
- 실질 작업 시간 외 문서 작업/일감 관리 시간이 추가됨
	- 진행 상황에 맞춰 관리 툴을 업데이트해줘야 한다.
		- 단 이것은 어떤 툴을 쓰느냐에 따라 손이 덜 갈 수 있다.

도입한다면 어떤 일감을 관리할 것인지 명확히 해야 한다. 또한 모든 일감을 관리할 것인지, 관리한다면 어떻게 관리할 것인지 확인해야 한다.
현재, 그리고 추후 관리해야 할 것으로 예상되는 일감의 종류는 다음과 같다.

- 기획
	- 기획안에 대한 일감은 생성할 수 있으나 상세 내용은 다른 방식으로 공유가 필요
- 디자인
	- 디자인에 대한 일감은 생성할 수 있으나 상세 내용은 다른 방식으로 공유가 필요
- 개발
	- 일감을 필요 단위로 나누고, 공유 가능
- QA
	- 대분류 단위로 일감 생성은 가능하나 상세 내용은 액셀등 문서를 통해 추가 공유가 필요
- 버그 픽스
	- 버그 상황, 진행 상태 등 공유 가능

즉 정리하자면

- 개발/QA/버그 일감 관리를 하고자 한다면 적합하다.
- 기획/디자인은 단순 일감 관리로 힘들 수 있다.

또한 단순히 일감 관리 툴을 쓰는 것이 아니라 관리 툴을 적절하게 쓰고 업무 프로세스에 확실하게 녹여내는 것이 중요하다.

- 예를 들어, 커밋/mr(pr)단위로 일감 업데이트 하는 것을 루틴화
- 버그/stuck 등이 발생시 즉시 메신저로 통보/공유 되는 시스템 구축
- 관련 문서 등이 생기면 업무 툴에 업데이트 하는 습관을 들인다.
	- 혹은 자동화 시스템 구축



## 일정 관리

마찬가지로 도입할 이유와 도입하지 않을 이유를 선정한다.

도입할 이유는 다음과 같다.

- 서로 일정을 조율하고, 확인하기에 용이하다. (휴가 등)
	- 이를 통해 앞으로의 일감 관리도 일정을 고려하기 편해진다.
- 정기 일정(주간 회의 등)이 포함된 경우, 잊지 않게 알람을 보내줄 수 있음
- 추가 일정을 잡더라도 어디에서 어떻게 얼마나 할지에 대한 관리가 편해진다.
	- 그냥 팀원들에게 일정 만들어서 참석 시키면 된다

도입하지 않을 이유는 다음과 같다.

- 일정이 생기면 업데이트 해줘야 한다.
- 회의 같은 경우 그냥 메신저로 잡는게 편할수도 있음.

마찬가지로 도입한다면 업무 프로세스에 확실하게 녹여내야 한다.

- 일정이 생긴다면 반드시 공유 툴에 업데이트
- 회의 등의 일정을 잡을 때도 단순히 메신저로 하는 것에 그치지 않고 툴에도 업데이트

## 문서 관리

도입할 이유는 다음과 같다.

- 문서를 한 곳에 저장해 두었기에 찾고 싶은 문서를 찾고 싶으면 한 곳만 뒤지면 된다
- 유지 보수 측면에서도 상세히 작성된 문서는 도움이 된다.
- 신규 인원이 투입된 경우에도 찾아볼 수 있는 레퍼런스가 생김
- 문서를 통해 일정/일감 등에 대한 자세한 공유가 가능
- 회의/업무 공유 등에서도 문서를 통해 모두 같은 레퍼런스를 공유가능

도입하지 않을 이유는 다음과 같다.

- 관리가 힘듦
- 명확한 규칙 없이는 문서가 파편화 될 가능성이 존재한다.
- 문서 관리를 위해 추가로 시간을 들여야 한다.
- 실 상황과 다르거나 제대로 관리되지 않은 문서는 오히려 업무에 악영향을 끼친다

도입한다면 마찬가지로 다음과 같은 사실들을 유념해 두어야 한다

- 문서 관리 규칙을 확립
	- 개인 문서/팀 문서 등 카테고리라이징을 명확히 하고, 문서의 폴더 구조를 통일 시거나, 규칙 문서를 최 상단에 두어 헷갈리거나 헤매지 않게 한다.
	- 문서 구조는 우선 개인이 알아서 작성하되, 적절한 문서 포맷이 생겨난다면 템플릿화
- 문서의 업데이트/최종확인 일자 등을 반드시 명시
	- 문서가 최신화 되지 않은 문서인지 확실하게 알 수있도록
	- 혹은 deprecated 한 문서라면 삭제하거나 명시
- 문서/일감을 동기화
	- 앞서 언급한 것 처럼 일감과 연계되는 문서는 항상 일감의 상태 변화에 동기화 되있어야 한다.
	- 개인 문서는 업데이트가 더디더라도 공용 문서의 업데이트는 항상 현재 서비스 상태와 동기화 되어있어야 한다.


## 의사소통

메신저는 기존에 사용하는 툴(teams)을 바꿔야 하는 이유를 설명한다. 즉, teams와 slack의 비교를 골자로 한다.

|         | Teams                        | Slack                                           |
| ------- | ---------------------------- | ----------------------------------------------- |
| 생태계     | Microsoft 친화적                | Saas 대부분에 친화적                                   |
| 가격      | Microsoft 365 구독중이라면 포함되어있음  | 별도 구매. 단, 소규모 팀이라면 무료                           |
| Webhook | Microsoft 생태계가 아니라면 어려울 수 있음 | 간단한 연동/알림 봇 생성 용이                               |
| 채팅      | 채팅/화상 중심, 서식이 망가지기 쉬움        | 스레드 중심, 채팅 흐름을 방해하지 않고 스레드 형식으로 논의가능, 대부분 서식 지원 |
| 파일 공유   | One drive에 종속됨               | 가능                                              |
|         |                              |                                                 |

## 무엇보다도

 협업 툴을 도입하여 써보는 것은 **일경험 인턴** 이라는 취지에 무엇보다도 맞는 경험으로 보임. 협업 툴을 도입하고, 기초부터 규칙을 도입하며 쓰는 경험은 인턴에게 있어 큰 도움이 되는 경험이 될 것으로 생각된다.
