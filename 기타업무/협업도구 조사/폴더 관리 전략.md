
## 전략 1. 지식 허브 구조

```markdown

Team HQ
├─ Home
├─ Product / Business Wiki
├─ Engineering Wiki
├─ Meeting Notes
├─ Decision Log
├─ Project Briefs
├─ Repos & Systems
├─ Templates
└─ Archive

```

우선 Notion을 오직 지식 저장소로만 쓰는 형태. 다른 업무 툴과 충돌하지 않기 때문에 협업 툴로 인한 업무 부하가 가장 적은 형태.


## 전략 2. 프로젝트 허브 구조

```markdown
Projects DB
├─ 프로젝트 A
│  ├─ 목표
│  ├─ 배경
│  ├─ 범위 / 제외 범위
│  ├─ 주요 의사결정
│  ├─ 관련 ClickUp List / Task 링크
│  ├─ 관련 GitLab Repo / MR 링크
│  └─ 산출물 / 회고
├─ 프로젝트 B
└─ 프로젝트 C
```

프로젝트의 **상위 맥락** 관리에 가장 적합한 구조. 프로젝트의 의사결정, 일감 관련 정보를 관리하기에 가장 최적화된 구조. 
프로젝트 히스토리 추적에도 장점이 있다.

## 전략 3. 개발 문서 중심 구조

Notion을 개발팀 문서 저장소로 쓰는 형태. 코드 변경 이력은 Gitlab에 두고, Notion에는 설계와 운영 문서를 둔다.


```markdown
Engineering Wiki
├─ Architecture
├─ API Docs
├─ DB Schema Notes
├─ Infra / Deployment
├─ Runbooks
├─ Incident Notes
├─ Coding Convention
├─ Git / Branch Strategy
├─ Release Process
└─ Repository Index
```
